## 5. 동일 음원 테스트 대비 비교

`wav/google_tts.wav` (14s 단일 음원 반복) vs 다중 음원 (라운드 평균 기준):

| 지표 | 동일 파일 50 | 다중 파일 50 | 동일 파일 100 | 다중 파일 100 |
|------|------------|------------|-------------|-------------|
| 총 오디오량 | 700s | 251.5s | 1,400s | 493.0s |
| 소요 시간 | 3.412s | 1.8–2.0s | 6.568s | 2.7–4.5s |
| RTF | 205× | 124–140× | 213× | 109–183× |
| avg latency | 0.265s | 0.370–0.433s | 0.520s | 0.619–0.823s |
| GPU 최대 util | 100% | 100% | 100% | 89–91% |

### RTF가 낮아진 이유

1. **클립이 짧다** (avg 5s vs 14s) → 청크 수가 적어 세션이 빠르게 완료 → 배칭 기회 감소
2. **길이 불균일** → 짧은 파일이 먼저 끝나면서 배치 크기가 동적으로 줄어듦
3. 소요 시간 자체는 짧음 — **단위 시간당 처리량은 충분**

### 100 세션에서 GPU 100% 미도달 원인

세션마다 다른 길이의 음원 → 짧은 파일 완료 후 배치 크기 축소 → GPU가 100%에 도달하지 못함.
50 세션은 균일한 5s 클립이라 모든 세션이 거의 동시에 같은 청크 구간을 처리하여 완전한 배치 형성 가능.

## 6. 결론

| 지표 | 50세션 R1 | 50세션 R2 | 100세션 R1 | 100세션 R2 |
|------|---------|---------|---------|---------|
| 소요 시간 | 1.8s | 2.0s | 4.5s | **2.7s** |
| RTF | 139.6× | 124.0× | 108.8× | **182.7×** |
| avg latency | 0.370s | 0.433s | 0.823s | **0.619s** |
| max latency | 0.441s | 0.595s | 0.985s | **0.782s** |
| 성공률 | 100% | 100% | 100% | 100% |

1. **50 세션은 안정적으로 재현**: 두 라운드 모두 2s 이내 완료, avg latency 0.5s 이하 — 신뢰도 높음.

2. **100 세션은 실행 조건에 따라 편차 존재**: 소요 시간 2.7s(R2)~4.5s(R1). TCP 커넥션 warm-up 여부가 주요 변수로, R2가 best case, R1이 worst case에 해당.

3. **worst case(R1) 기준에서도 실용 수준 유지**: 최대 latency 0.985s로 실시간 응답 기준(1s 이내) 충족.

4. **GPU 포화는 배치 균일성에 의존**: 다중 음원 100 세션에서 GPU가 89–91%에 머무는 것은 불균일 배치 때문이며, 이는 구조적 특성으로 최적화 여지 있음.
