# 동시 WebSocket 스트리밍 부하 테스트 (1 / 10 / 30 / 50 / 80 / 100 세션)

- **날짜**: 2026-05-22
- **서버**: `ws://172.31.79.202:30000/api/ws` (FastAPI + vLLM `AsyncLLMEngine`)
- **테스트 스크립트**: `tests/test_asr_ws_concurrent_load.py`
- **음원**: `wav/sequential/` 정렬 첫 100개 한국어 음원 (경동나비엔 콜센터 시나리오)
- **송신 모드**: **실시간 페이싱** — 200ms 청크를 wall-clock에 맞춰 전송 (실 사용자 환경 시뮬레이션)
- **워밍업**: 측정 전 sequential 3회 (cold-start 회피)
- **레벨 간 settle**: 4초

## 1. 측정 지표 정의

| 지표 | 정의 |
|------|------|
| **TTFT** | 세션 시작(t0, ready 수신 직후) → 첫 non-empty `result(partial)` 수신까지 (ms) |
| **end→final** | `{"type":"end"}` 송신 시각 → `result(is_final=true)` 수신까지 (ms) |
| **RTF/session** | `wall_clock / audio_duration` (1.0 = 실시간 유지) |
| **thru_RTF** | `Σ audio_duration / level_wall_clock` (×배율, 처리량 관점) |
| **chunk_lag** | partial 도착 시각 − 해당 chunk의 audio_pos (음수 = 실시간 대비 앞섬, 큰 양수 = 서버가 못 따라감) |

- 전 레벨 **성공률 100%** (총 271 세션 + 워밍업 3 = 274 세션 전부 성공)

## 3. 지표별 해석

### 3-1. TTFT (첫 partial 도착)

| sess | min | avg | p50 | p95 | max |
|----:|----:|----:|----:|----:|----:|
| 1   |  847 |  847 |  847 |  847 |  847 |
| 10  |  886 |  905 |  908 |  917 |  917 |
| 30  |  938 |  957 |  959 |  972 |  979 |
| 50  |  958 |  988 |  991 | 1014 | 1025 |
| 80  | 1101 | 1271 | 1289 | 1356 | 1388 |
| 100 | 1102 | 1164 | 1159 | 1254 | 1311 |

- 1 sess 기준 847ms ≈ 서버 chunk_size_sec(1.0초) + 모델 추론 시간. **이론적 하한**.
- 50 sess까지는 +140ms 이내로 안정적 증가 → 배치 큐 영향 적음.
- 80 sess부터 +400ms 증가 → 동시 첫-청크 배치 크기 증가에 따른 추론 latency.

### 3-2. end→final (음원 종료 후 최종 응답)

| sess | min | avg | p50 | p95 | max |

| 1   |  48 |  48 |  48 |  48 |  48 |

## 4. 1차 측정에서 관찰된 cold-start 이슈 (해소됨)

워밍업 1회만 두고 첫 cold 실행 시 **80 sess만 TTFT 5,200ms로 스파이크**한 1회성 현상 발생:

| run | warmup | 80 sess TTFT_avg | 80 sess lag_avg | RTFs p95 |
|----|----:|----:|----:|----:|
| 1차 (cold) | 1회 | **5200ms** | **+2849ms** | **1.818** |
| 2차 (warm) | 1회 | 1086ms | +73ms | 1.063 |
| 3차 (warm) | 1회 | 1111ms | +95ms | 1.064 |
| 4차 (현 기본값) | **3회** | 1271ms | +181ms | 1.131 |

### 원인 분석

| 후보 | 검증 결과 |
|------|---------|
| 구조적 80-sess 임계점 | ❌ 2, 3, 4차에서 재현 안 됨. 100 sess가 80 sess보다 항상 약간 더 무거움 — 임계점이면 매 run 재현돼야 함 |
| 레벨 간 4초 쿨타임 부족 | ❌ stable run에서 chunk_lag가 모든 레벨 +200ms 이내, 큐 누적 없음. 쿨타임을 늘리면 cold-start 리스크만 증가 |
| **vLLM AsyncEngine / KV 캐시 cold-start** | ✅ 가장 유력. 80 sess의 첫-청크가 cold 배치를 만나 5초 처리, 그 동안 80 chunk가 모두 queue → 한 배치로 동시 emit. TTFT 분포 5104~5323ms로 ±220ms 매우 좁은 클러스터인 점이 단서 |
| 네트워크/스크립트 jitter | ❌ jitter면 TTFT 분산이 컸을 것. 좁은 클러스터와 모순 |

### 적용한 보강

`WARMUP_ROUNDS = 3` (sequential 워밍업 3회). 측정 전 모델/스케줄러/KV 캐시를 미리 데워 cold 배치 회피.

```
[warmup] 3 sequential sessions
  [1] ok  TTFT=910ms   end→final=+58ms  RTF_sess=1.009   ← cold (1회차)
  [2] ok  TTFT=1112ms  end→final=+71ms  RTF_sess=1.011
  [3] ok  TTFT=866ms   end→final=+52ms  RTF_sess=1.008   ← warm 도달
```

## 5. 결론

1. **동시 100 세션까지 100% 성공률** + p95 TTFT 1.25초, p95 end→final 0.21초로 **실시간 대화 응답 기준 충족**.
2. RTF/session p95 = 1.075 → 서버는 100 세션에서도 실시간 페이싱에 7.5% 여유. 더 높은 동시성 여력 있음.
3. **cold-start 이슈는 측정 방법의 함정**이지 서버 결함이 아님. 워밍업 3회로 안정적 측정 가능.
4. 80 vs 100 sess 차이가 작음 (TTFT 1271 vs 1164ms) → vLLM 연속 배칭이 잘 작동하고 있음을 시사. 다음 측정 시 150/200 세션까지 확장해 실제 saturate 지점 탐색 권장.

## 6. 한계 및 후속
