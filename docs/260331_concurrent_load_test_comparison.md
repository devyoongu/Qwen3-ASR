# Concurrent ASR 부하 테스트 비교 보고서 (10 / 50 / 100 / 150 동시 세션)

- **날짜**: 2026-03-31
- **서버**: `http://172.31.79.202:30000` (FastAPI + vLLM `AsyncLLMEngine`)
- **테스트 스크립트**: `examples/example_qwen3_asr_vllm_streaming_concurrent.py`
- **테스트 오디오**: `wav/google_tts.wav` (한국어, ~14초, 1000ms 청크 × 14회 + finish 1회 = 15 API calls/세션)

| 동시 세션 | 총 오디오량 | 전체 소요 시간 | 최초 latency min | avg | max | 성공률 |

## 3. 확장성 분석

### 3-1. 처리 시간 스케일링

| 세션 수 비율 | 소요 시간 비율 | 평가 |
|------------|--------------|------|
| 10 → 50 (5×) | 1.899 → 3.412 (**1.80×**) | 배치 효율 향상 (36%) |
| 50 → 100 (2×) | 3.412 → 6.568 (**1.93×**) | 거의 선형 (96%) |
| 100 → 150 (1.5×) | 6.568 → 9.657 (**1.47×**) | 거의 선형 (98%) |
| 10 → 150 (15×) | 1.899 → 9.657 (**5.09×**) | 배치 효율로 인한 절감 |

→ **전체 소요 시간이 세션 수에 거의 정비례** — continuous batching의 선형 확장성 확인.

### 3-2. 처리량 (Throughput)

| 동시 세션 | 실시간 배율 (RTF) | 세션 처리량 |
|-----------|----------------|-----------|
| 10 | 74× | 5.3 세션/s |
| 50 | 205× | 14.7 세션/s |
| 100 | 213× | 15.2 세션/s |
| 150 | 217× | 15.5 세션/s |

→ 10 세션에서 74×에 불과하던 RTF가 50 세션 이상에서 **200×대로 급증** — 배치가 충분히 찰수록 GPU 효율이 크게 향상.

### 3-3. 최초 응답 Latency 증가

| 동시 세션 | avg latency | 단일 세션 대비 배수 |
|-----------|------------|-----------------|
| 1 (참고) | ~0.12s | 1× |
| 10 | 0.161s | **1.3×** |
| 50 | 0.265s | **2.2×** |
| 100 | 0.520s | **4.3×** |
| 150 | 0.839s | **7.0×** |

→ latency는 세션 수보다 빠르게 증가하지 않음. 150 세션에서도 평균 **0.839s** 로 실시간 대화 수준의 응답 속도 유지.

## 4. 최초 응답 Latency 분포

| 구간 | 10 세션 | 50 세션 | 100 세션 | 150 세션 |
|------|---------|---------|---------|---------|
| < 0.3s | **10 (100%)** | **40 (80%)** | 5 (5%) | 3 (2%) |
| 0.3 – 0.5s | 0 | 10 (20%) | 30 (30%) | 4 (3%) |
| 0.5 – 0.8s | 0 | 0 | 61 (61%) | 34 (23%) |
| 0.8 – 1.5s | 0 | 0 | 4 (4%) | **109 (73%)** |
| ≥ 1.5s | 0 | 0 | 0 | 0 |

- **10 세션**: 전원 0.3s 이내 (0.157–0.165s) — 단일 세션과 거의 동일한 응답 속도
- **50 세션**: 전원 0.5s 이내, 80%가 0.3s 이내 응답 — 매우 쾌적
- **100 세션**: 95%가 0.8s 이내, tail latency 1.161s
- **150 세션**: 전원 1.5s 이내, 최대 0.944s — tail latency 안정적

## 5. GPU 사용률 분석

### 5-1. GPU 100% Util 구간

| 동시 세션 | GPU 100% 시작 | GPU 100% 종료 | 지속 시간 |
|-----------|-------------|-------------|---------|
| 10 | 10:46:45.348 | 10:46:46.250 | **0.9s** |
| 50 | 10:48:03.228 | 10:48:06.133 | **2.9s** |
| 100 | 10:31:23.423 | 10:31:29.333 | **5.9s** |
| 150 | 10:49:38.054 | 10:49:46.967 | **8.9s** |

→ GPU 100% 지속 시간도 세션 수에 거의 정비례.

### 5-2. 세션당 GPU 연산 시간

| 동시 세션 | GPU 100% 시간 | 세션당 GPU 시간 |
|-----------|-------------|--------------|
| 10 | 0.9s | 0.090s |
| 50 | 2.9s | **0.058s** |
| 100 | 5.9s | **0.059s** |
| 150 | 8.9s | **0.059s** |

→ 50개 이상부터 세션당 GPU 연산 시간이 **0.058–0.059s로 수렴** — 배치가 충분히 클 때 GPU가 최적 효율로 동작.
→ 10 세션은 배치 크기가 작아 상대적으로 효율이 낮음 (0.090s/세션).

### 5-3. GPU 파워 및 메모리 대역폭

| 동시 세션 | 피크 파워 | 메모리 대역폭 (100% 구간) | 유휴 파워 |
|-----------|---------|----------------------|---------|
| 10 | 323W | **78%** | ~21W |
| 50 | 323W | 66–70% | ~21W |
| 100 | 322W | 64–80% | ~22W |
| 150 | 323W | 44–70% | ~22W |

- 피크 파워는 모든 케이스에서 **322–323W 로 일정** (GPU TDP 내 정상 동작)
- 메모리 대역폭은 배치 구성에 따라 변동; 150 세션에서 초반 44%로 시작 후 70%까지 상승 (배치 파이프라인 warm-up 효과)
- GPU VRAM 사용량: 모든 테스트에서 **19,689 MiB 고정** (vLLM 시작 시 pre-allocate)
- PCIe 링크: 유휴 Gen1 → 추론 시 Gen3 자동 전환

## 6. 전체 결론

| 지표 | 10 세션 | 50 세션 | 100 세션 | 150 세션 |
|------|---------|---------|---------|---------|
| 전체 소요 시간 | 1.9s | 3.4s | 6.6s | 9.7s |
| 오디오 처리량 (RTF) | 74× | 205× | 213× | 217× |
| 평균 최초 latency | 0.161s | 0.265s | 0.520s | 0.839s |
| tail latency (max) | 0.165s | 0.464s | 1.161s | 0.944s |
| GPU 100% Util 시간 | 0.9s | 2.9s | 5.9s | 8.9s |
| 성공률 | 100% | 100% | 100% | 100% |

### 핵심 결론

1. **선형 확장**: 세션 수가 3× 증가(50→150)해도 처리 시간은 2.83× 증가 — 배치 효율이 유지되며 거의 선형 확장.

2. **GPU 포화 임계점**: 10 세션(0.090s/세션)에서 50 세션(0.058s/세션)으로 넘어가는 구간에서 GPU 효율이 급격히 향상, 이후 100·150 세션에서 ~0.059s/세션으로 수렴. 약 50 세션 이상이 GPU 최적 동작 포인트.

3. **latency 안정성**: 150 세션에서 최대 latency 0.944s로, `max_tokens=32` 제한과 continuous batching 덕분에 꼬리 latency가 급증하지 않음.

4. **콜센터 운용 가능 수준**: 150 동시 통화에서 평균 0.839s 첫 응답 — 실시간 ASR 요구 조건(1초 이내) 충족.
