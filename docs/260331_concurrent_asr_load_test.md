## 3. 최초 응답 Latency 분포

| 구간 | 세션 수 | 비율 |
|------|---------|------|
| < 0.3s | 5 | 5% |
| 0.3s – 0.5s | 30 | 30% |
| 0.5s – 0.8s | 61 | 61% |
| ≥ 0.8s | 4 | 4% |

- 전체의 **95%가 0.8s 이하** 에서 첫 번째 텍스트를 수신
- 최상위 배치(batch)에 할당된 워커(5개)는 단일 세션 수준(0.211s)과 거의 동일한 응답 속도
- 꼬리(tail) latency는 1.161s 로 단일 세션 대비 약 10× 수준 — 100개 요청 동시 처리를 고려하면 합리적인 범위

## 4. 단일 세션 대비 비교

| 지표 | 단일 세션 (1000ms 청크) | 100 동시 세션 |

| **Speedup** | - | **~30×** 이상 |

> 단일 세션 latency 0.119s 는 `docs/streaming_asr_test_result.md` 기준.
> 직렬 처리 시간은 단일 세션 소요 시간(~2s) × 100으로 추산.

**핵심**: `AsyncLLMEngine`의 continuous batching으로 100개 세션이 GPU를 공유하면서도
평균 latency 증가가 0.119s → 0.520s(4.4×)에 그쳤고, 전체 처리 시간은 직렬 대비 ~30× 단축.

## 5. GPU 사용률 분석

모니터링 명령:
```bash
nvidia-smi --query-gpu=timestamp,power.draw,utilization.gpu,utilization.memory,memory.used,pcie.link.gen.current,pcie.link.width.current \
  --format=csv -lms 100 -f gpu_log.csv
```

### 5-1. 타임라인

| 구간 (KST) | GPU Util | Power | 상태 |
|------------|----------|-------|------|
| ~10:31:22.3 이전 | 0% | ~22W | 서버 대기 (유휴) |
| 10:31:22.4 | 77% | 58W | 첫 번째 배치 처리 시작 |
| 10:31:22.9 | 77% | 130W | 배치 규모 증가 |
| **10:31:23.4 – 10:31:29.3** | **100%** | **298–322W** | **Continuous batching 풀 가동** |
| 10:31:29.3 – 10:31:45 | 0% | 260W → 106W | 추론 완료, power cooldown |
| 10:31:47 이후 | 0% | ~22W | 유휴 복귀 |

### 5-2. 관찰 포인트

- **GPU 100% 지속 구간**: ~5.9초 — 1,400초 분량의 오디오를 처리하는 동안 GPU가 쉬지 않음
- **메모리 사용량**: 19,689 MiB 고정 (vLLM 시작 시 pre-allocate, 추론 중 변동 없음)
- **PCIe 링크**: 유휴 시 Gen1 → 추론 시 Gen3 자동 전환 (전력 관리)
- **메모리 대역폭**: 64–80% (100× 배치 처리 중 충분히 활용)
- **피크 전력**: 322W (GPU TDP 내 정상 동작)

```
Power 프로파일:
  유휴: ~~~~22W~~~~
  증가: 22→58→130→298→322W  (약 1초에 걸쳐 램프업)
  추론: 322W 평탄 구간 (5.9s)
  냉각: 322→260→106→58→23→22W (약 18초에 걸쳐 스텝다운)
```

## 6. 인식 정확도

100개 세션 전원 동일한 음성을 처리하여 동일한 최종 결과 반환:

```
안녕하세요. 포지큐브 고객센터 AI 콜봇입니다. 보다 정확한 상담을 위해
조용한 곳에서 상담사와 이야기 하듯이 편하게 말씀해 주시면 더 빠르게
도와드릴 수 있습니다. 무엇을 도와드릴까요?
```

- 동시 부하에서도 인식 오류 없음 (세션 간 state 격리 정상 동작)
- 스트리밍 중간 결과도 모든 워커에서 일관된 진행 패턴 확인
- `asyncio.Lock` 기반 세션 내 직렬화가 올바르게 동작 (청크 순서 뒤집힘 없음)

## 7. 결론

| 지표 | 결과 |
|------|------|
| 100 동시 세션 처리 | **성공 (100/100)** |
| GPU continuous batching | **동작 확인** (6s간 100% Util 유지) |
| 평균 최초 응답 latency | **0.520s** (단일 대비 4.4×) |
| 직렬 처리 대비 speedup | **~30×** 이상 |
| 처리량 | **1,400초 오디오 / 6.568s** = 실시간 **213×** |

`FastAPI + AsyncLLMEngine` 구조는 100개 동시 ASR 세션 부하에서 안정적으로 동작하며,
vLLM continuous batching을 통해 GPU를 최대한 활용한다.
콜센터 등 다수 동시 통화 처리 시나리오에 적합한 성능임을 확인.
