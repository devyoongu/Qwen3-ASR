### 인식 정확도

**공통 오류**:
- `조용한 곳에서` → `조용한 공간에서` (모든 step에서 발생)
- 중간 결과에서 `말씀해 주시죠` / `이야기하듯.` 등 의역 발생 후 수정됨

**특수 문자 이슈** (이전 실행):
- 스트리밍 중 `AI 콜봇` 사이에 `AI  콜봇` 형태로 `\ufffd` (U+FFFD, Replacement Character) 문자 발생
- 클라이언트 후처리 (`.replace('\ufffd', '')`) 로 해결

### \ufffd 발생 원인 상세

#### 배경: 한글 UTF-8 인코딩

한글 한 글자는 UTF-8에서 **3바이트**로 인코딩된다.

```
'콜' → 0xEC 0xBD 0x9C  (3바이트)
'봇' → 0xEB 0xB4 0x87  (3바이트)
```

#### 발생 메커니즘

LLM은 텍스트를 **토큰(token)** 단위로 생성한다. 스트리밍 모드에서는 토큰이 생성될 때마다 중간 결과를 즉시 반환하는데, 이 과정에서 다음 문제가 발생한다.

```
[LLM 내부 토큰 생성]
  토큰 N   → bytes: 0xEC 0xBD        (2바이트, '콜'의 앞부분)
  토큰 N+1 → bytes: 0x9C 0xEB 0xB4  (나머지 + '봇' 앞부분)

[중간 결과 반환 시점 - 토큰 N만 있을 때]
  bytes: ...0xEC 0xBD  → UTF-8 디코딩 시도
  → 0xEC 0xBD 는 불완전한 시퀀스 (3바이트가 필요한데 2바이트만 존재)
  → Python/서버가 decode(errors='replace') 처리
  → '\ufffd' (=  ) 로 대체 출력
```

#### 왜 고착되는가

한번 `\ufffd`로 출력된 중간 결과가 이후 call에서도 계속 유지되는 이유는, 서버의 스트리밍 state가 **확정된 텍스트(fixed)** 와 **미확정 텍스트(unfixed)** 를 분리 관리하기 때문이다. 불완전하게 디코딩된 바이트가 확정 영역에 포함되면 이후 call에서도 그대로 유지된다.

```
call 009: 'AI 콜봇입니다.'          ← 정상 (아직 unfixed 구간)
call 010: 'AI  콜봇입니다. 보다...' ← \ufffd 고착 시작 (fixed 영역에 포함됨)
call 028: 'AI  콜봇입니다. ...'     ← 끝까지 유지
```

#### 근본 원인 vs 해결 위치

| 위치 | 방법 | 비고 |
|------|------|------|
| **서버 (근본 해결)** | partial 토큰 바이트 누적 후 완성된 문자만 디코딩 | `errors='ignore'` 또는 바이트 버퍼링 |
| **클라이언트 (적용됨)** | `.replace('\ufffd', '').strip()` | 깨진 문자 제거, 이중 공백은 남을 수 있음 |
