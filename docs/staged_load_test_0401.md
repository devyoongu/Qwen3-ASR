### 4-3. 100세션에서 GPU Util이 92%에 머문 원인

100세션 테스트에서 GPU가 **최대 92%**에 그친 반면 50세션은 100%, 150세션도 100%에 도달한 이유:

- 100개 파일의 **길이 분산이 크다** (min 3.3s ~ max 7.0s)
- 짧은 파일들(3~4s)이 먼저 완료 → 2번째 배치 이후 세션 수가 급감 → 배치 크기 축소
- 50세션은 파일들이 비슷한 길이대라 상대적으로 동기화가 잘 되어 균일한 배치 유지
- 150세션은 세션 수가 충분히 많아 일부 완료 후에도 배치가 일정 크기 이상 유지 → 후반 100% 도달

## 6. 결론

| 항목 | 50세션 | 100세션 | 150세션 |
|------|-------|--------|--------|
| 소요 시간 | 2.229s | 3.051s | 3.889s |
| RTF | 112.8× | 161.6× | **194.9×** |
| avg latency | 0.423s | 0.675s | 0.876s |
| max latency | 0.454s | 0.913s | **1.119s** |
| 1s 기준 충족 | ✅ (avg) | ✅ (avg) | ⚠️ avg 충족, max 초과 |
| 성공률 | 100% | 100% | 100% |

1. **150세션 처리 가능 확인**: 이전 예측(ValueError 발생)은 당시 음원 파일이 100개였을 때의 제약이었으며, 150개 파일 준비 후 정상 동작 확인.

2. **RTF는 세션 수에 비례해 상승**: 50(112.8×) → 100(161.6×) → 150(194.9×) — 세션이 많을수록 continuous batching 효율이 올라가 단위 시간당 처리량 증가.

3. **Latency는 세션 수에 비례해 증가**: avg 0.423s → 0.675s → 0.876s. 150세션 max latency가 1.119s로 1s 임계치를 소폭 초과하나, 71%는 0.8–1.5s 이내.

4. **GPU 포화 패턴이 비단조적**: 100세션(92%)보다 150세션(100%)에서 GPU가 더 높게 활용됨. 배치 크기가 음원 길이 불균일성의 영향을 받아 중간 세션 수에서 오히려 포화 효율이 낮아지는 현상.

5. **5초 대기 구간의 PCIe 상태 유지**: 단계 간 대기 중에도 PCIe Gen3가 유지되어 다음 단계 요청 시 즉시 추론 시작 가능. TCP 커넥션 warm 상태 지속의 이점.
